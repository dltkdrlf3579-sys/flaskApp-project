# 기술부채 로드맵

작성일: 2026-06-13  
범위: 0~9단계 정적 분석 결과를 바탕으로 한 운영 안정화 중심 로드맵. 코드 수정, 서버 실행, DB 접속, 테스트 실행은 하지 않음.

## 1. 한 줄 결론

이 프로젝트의 기술부채는 “낡은 코드” 자체보다, SQLite에서 PostgreSQL로 급히 넘어오고 여러 게시판 기능을 빠르게 붙이면서 생긴 실행 경로, DB 스키마, 프론트 API, 보드별 구현 방식의 불일치가 핵심이다.

따라서 지금 필요한 것은 대규모 재작성이나 보안 대공사가 아니라, 실제 사용자가 누르는 화면과 데이터가 깨지지 않도록 만드는 운영 안정화다.

## 2. 이번 로드맵의 기준

### 우선하지 않는 것

- 외부 서비스 수준의 보안 강화.
- SSO/CSRF/권한 엔진의 대규모 재설계.
- 작동 여부를 모르는 상태에서 `app.py` 대규모 분리.
- `pytest` 전체 실행으로 한 번에 검증하려는 시도.

### 우선하는 것

- 새 PC나 새 DB에서 실행했을 때 빠지는 테이블이 없게 만들기.
- 화면의 버튼이 실제 존재하는 API를 호출하게 만들기.
- 삭제, 업로드, import/export처럼 데이터에 영향을 주는 기능을 예측 가능하게 만들기.
- 게시판별로 같은 기능이 다르게 동작하는 부분을 줄이기.
- 리팩터링 전에 작은 검증 루틴부터 확보하기.

## 3. 기술부채 분류

| 영역 | 현재 상태 | 영향 | 우선도 |
|---|---|---|---|
| 실행 진입점 | `python app.py`, WSGI import, first request 초기화 경로가 서로 다른 부작용을 가짐 | 실행 방식에 따라 DB 초기화/동기화 타이밍이 달라질 수 있음 | 높음 |
| DB 마이그레이션 | numbered migration, 개별 setup script, 런타임 DDL이 공존 | 새 환경 구축 시 일부 테이블/컬럼이 누락될 수 있음 | 높음 |
| 프론트-API 연결 | 템플릿/JS가 존재하지 않거나 다른 이름의 API를 호출하는 사례가 있음 | 버튼 클릭 시 404, 기능 미동작, 잘못된 import/export 가능 | 높음 |
| 게시판 구조 | 동적 보드 엔진과 보드별 전용 Repository/Service가 섞여 있음 | 같은 기능도 보드마다 버그와 수정 위치가 다름 | 중간~높음 |
| 컬럼/섹션 관리 | 컬럼 서비스가 여러 계층에 중복되고 protected column 정책도 다름 | 컬럼 추가/삭제/표시 정책이 화면별로 달라질 수 있음 | 중간~높음 |
| `custom_data` | JSONB object와 JSON string을 모두 방어하는 코드가 많음 | 검색/렌더링/엑셀/마이그레이션에서 타입 문제가 날 수 있음 | 중간 |
| 테스트 자산 | `test_*.py`가 실제 테스트와 진단/수정 스크립트가 섞여 있음 | 전체 테스트 실행 자체가 위험할 수 있음 | 높음 |
| 루트 스크립트 | 수정, 점검, 마이그레이션, 임시 실험 파일이 섞여 있음 | 어떤 파일이 안전한지 판단하기 어려움 | 중간 |
| 설정/인코딩 | Windows Korean locale, PowerShell 5.1, UTF-8 Markdown이 섞임 | 한글 문서/체크리스트 수정 시 깨짐 가능 | 중간 |
| 권한/SSO | 여러 권한 엔진과 SSO debug 경로가 존재함 | 내부망 기준에서는 후순위. 단, 데이터 변경 API의 최소 보호는 운영 안전 관점에서 필요 | 낮음~중간 |

## 4. 1순위: 운영 사고 방지

### 4.1 실행/DB 기준선 고정

목표는 “어떻게 실행해야 안전한지”를 하나로 정하는 것이다.

- `python app.py` 직접 실행의 부작용과 WSGI import의 부작용을 문서 기준으로 구분한다.
- DB 백업 절차를 먼저 고정한다.
- `init_db()`가 실제로 실행하는 migration과 누락되는 setup script를 비교한다.
- 런타임에 테이블/컬럼을 만드는 코드 위치를 목록화한다.
- 새 환경 구축 순서를 하나의 체크리스트로 만든다.

관련 문서:

- `RUNTIME_ENTRY_REPORT.md`
- `SAFE_EXECUTION_PLAN.md`
- `PREFLIGHT_EXECUTION_REPORT.md`
- `DB_USAGE_MAP.md`

### 4.2 프론트-API 불일치 제거

목표는 “화면에서 누르는 버튼이 실제 API와 맞는지”를 먼저 맞추는 것이다.

- 존재하지 않는 API 호출을 실제 백엔드 라우트와 맞춘다.
- export/import/final-check 계열을 보드별로 표준화한다.
- 레거시 URL과 신규 URL 중 실제 사용할 것을 정한다.
- 403/error/login 관련 링크가 실제 존재하는 경로로 이동하는지 확인한다.

관련 문서:

- `ROUTE_MAP_REPORT.md`
- `FRONTEND_CONNECTION_REPORT.md`
- `BOARD_FLOW_REPORT.md`

### 4.3 위험한 테스트/스크립트 분리

목표는 “검증하려다 DB를 건드리는 사고”를 막는 것이다.

- `pytest` 전체 실행 금지를 유지한다.
- 순수 단위 테스트 후보와 DB/HTTP/write 후보를 분리한다.
- 진단 스크립트, 수정 스크립트, 마이그레이션 스크립트를 폴더별로 분리할 계획을 세운다.
- 안전한 smoke test 명령만 별도로 문서화한다.

관련 문서:

- `TEST_ASSET_REPORT.md`
- `SAFE_EXECUTION_PLAN.md`

## 5. 2순위: 구조 일관성 회복

### 5.1 게시판 공통 구조 정리

현재 보드들은 한 번에 다 합치면 위험하다. 우선 “어떤 보드가 어떤 계층을 쓰는지”부터 고정해야 한다.

권장 순서:

1. `follow_sop`과 `subcontract` 계열처럼 이미 동적 보드 엔진을 공유하는 그룹을 먼저 안정화한다.
2. `full_process`, `safe_workplace`의 forked Repository를 비교한다.
3. `accident`, `safety_instruction`처럼 독자 구조가 강한 보드는 마지막에 정리한다.
4. `change_request`, `partner`는 레거시 성격이 강하므로 별도 분리 대상으로 둔다.

### 5.2 컬럼/섹션/코드 관리 단일화

컬럼 관련 기술부채는 사용자 화면과 DB 스키마를 동시에 건드릴 수 있으므로 조심해야 한다.

권장 순서:

1. protected column 목록을 하나로 통합한다.
2. physical column과 JSONB-only column 기준을 보드별로 표로 고정한다.
3. `column_service.py`, `board_services.ColumnService`, `column_config_repository.py` 중 표준 진입점을 정한다.
4. 런타임 `ALTER TABLE`은 migration으로 옮기는 방향을 잡는다.

### 5.3 `custom_data` 타입 정리

목표는 DB 내부 기준을 JSONB object로 고정하고, 문자열 변환은 입출력 경계에서만 처리하는 것이다.

권장 순서:

1. 현재 `custom_data`를 쓰는 테이블과 화면을 고정한다.
2. 저장 시점의 타입을 하나로 통일한다.
3. 렌더링/엑셀/검색에서 방어적 파싱을 줄인다.
4. 기존 데이터 보정은 별도 migration으로 다룬다.

## 6. 3순위: 리팩터링 후보

| 후보 | 이유 | 선행 조건 |
|---|---|---|
| `change_request` Controller/Repository 분리 | `app.py` 내부 결합도가 높음 | route/API drift 정리 |
| `partner` 계열 분리 | 협력사, 첨부, 자동업로드, 검색이 넓게 퍼짐 | 파일/첨부 흐름 안정화 |
| export/import 공통화 | 보드별 호출 경로와 처리 방식이 다름 | 프론트-API 경로 확정 |
| Board Repository 공통화 | 같은 CRUD/검색/첨부 기능이 보드별로 다름 | 보드별 차이표 완성 |
| 설정 로더 정리 | 로컬/운영/비밀값이 섞임 | 실행 기준선 확정 |
| 테스트 체계 재구성 | 리팩터링 안전망 필요 | 안전 테스트 후보 분리 |
| 권한 엔진 정리 | 내부망 기준 후순위이나 장기 유지보수에는 필요 | 데이터 변경 API 정책 확정 |

## 7. 하지 말아야 할 순서

다음 순서는 피하는 것이 좋다.

1. `app.py`를 먼저 대규모로 쪼갠다.
2. 그 다음 실행해본다.
3. 깨진 화면을 하나씩 고친다.
4. 마지막에 DB migration을 맞춘다.

이 방식은 작년의 “뭔가 하는 척했는데 실행하면 다시 오류” 패턴으로 돌아갈 가능성이 높다.

## 8. 권장 순서

권장 순서는 다음과 같다.

1. 백업과 실행 기준선을 고정한다.
2. 전체 테스트 실행 금지를 유지하고 안전 검증만 분리한다.
3. 프론트-API 불일치를 먼저 고친다.
4. DB migration 누락과 런타임 DDL을 정리한다.
5. 보드별 공통/차이표를 기준으로 작은 단위 리팩터링을 한다.
6. 리팩터링마다 최소 smoke test를 붙인다.
7. 내부망 기준의 최소 데이터 보호만 남기고 보안 대공사는 하지 않는다.

## 9. 결론

이 프로젝트의 다음 목표는 “멋진 구조”가 아니라 “실행 가능한 안정성”이다.

가장 좋은 첫 코드 수정 대상은 큰 구조 변경이 아니라, 사용자가 실제로 누르는 화면과 API의 불일치, 새 DB에서 빠지는 schema, 그리고 위험한 테스트/스크립트 실행 방지다.
